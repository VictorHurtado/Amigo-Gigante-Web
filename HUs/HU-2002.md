# HU-2002 — Revisión visual del componente de búsqueda/filtro

**Como** usuario  
**Quiero** que el componente de búsqueda/filtro se vea visualmente atractivo y profesional  
**Para** tener una mejor experiencia al buscar animales

---

## Dependencias

- Componente de búsqueda/filtro existente (si aplica)
- HU-008 (Design System centralizado)
- HU-010 (Atomic Design - componentes reutilizables, si aplica)

---

## Contexto / Notas

- Esta HU corrige problemas visuales del componente de búsqueda/filtro.
- El componente actual no se ve visualmente lindo según feedback.
- El componente debe incluir:
  - Filtro de tipo de animal (ej: "Perros")
  - Filtro de ubicación (ej: "Cualquier ubicación")
  - Botón de búsqueda
- Se debe mejorar la apariencia visual manteniendo la funcionalidad.
- Debe seguir el Design System y ser consistente con el resto de la aplicación.

---

## Alcance

Incluye:
- Revisar el componente de búsqueda/filtro actual
- Mejorar el diseño visual (colores, espaciados, bordes, sombras)
- Asegurar que use tokens del Design System
- Mejorar la jerarquía visual
- Ajustar tamaños y proporciones
- Mejorar estados hover y focus
- Asegurar que sea responsive

No incluye:
- Cambios en la funcionalidad
- Agregar nuevos filtros
- Cambios en la lógica de búsqueda
- Cambios en otras pantallas

---

## Criterios de aceptación (Given / When / Then)

### 1) Diseño visual mejorado

- **Dado** que visualizo el componente de búsqueda/filtro
- **Cuando** observo su apariencia
- **Entonces** el componente se ve visualmente atractivo
- **Y** tiene una jerarquía visual clara
- **Y** los elementos están bien alineados
- **Y** los espaciados son consistentes

---

### 2) Uso del Design System

- **Dado** que se mejora el componente
- **Cuando** se revisa el código
- **Entonces** se usan tokens del Design System:
  - Colores: `brand-500`, `accent-500`, `neutral-*`
  - Espaciados: tokens de spacing
  - Bordes: `rounded-lg` o similar
  - Sombras: tokens de sombra
- **Y** no hay colores hardcodeados fuera de los tokens

---

### 3) Estados interactivos mejorados

- **Dado** que interactúo con el componente
- **Cuando** hago hover sobre elementos
- **Entonces** hay feedback visual claro
- **Y** los estados de focus son visibles
- **Y** las transiciones son suaves
- **Y** el botón de búsqueda tiene estado hover destacado

---

### 4) Iconos y elementos visuales

- **Dado** que el componente tiene iconos
- **Cuando** visualizo el componente
- **Entonces** los iconos están bien posicionados
- **Y** tienen tamaño apropiado
- **Y** tienen color consistente con el Design System
- **Y** están alineados con el texto

---

### 5) Responsividad

- **Dado** que el componente es responsive
- **Cuando** cambio el tamaño de la pantalla
- **Entonces** el componente se adapta correctamente
- **Y** los elementos se reorganizan apropiadamente en móvil
- **Y** mantiene buena apariencia en todos los tamaños

---

### 6) Consistencia con el resto de la app

- **Dado** que se mejora el componente
- **Cuando** se compara con otros componentes
- **Entonces** mantiene consistencia visual
- **Y** usa los mismos patrones de diseño
- **Y** sigue las convenciones establecidas

---

## Reglas técnicas

### Arquitectura
- Presentation solo consume UseCases vía IoC (no importa repositorios concretos).
- Domain no importa React/Next/MUI/Tailwind/Inversify.
- Infrastructure contiene implementaciones y wiring IoC.
- **Solo cambios en Presentation**: componentes y estilos.

### UI
- Usar MUI para componentes base.
- Tailwind se usa para layout/spacing/responsive utilities.
- Usar tokens del Design System (`brand-500`, `accent-500`, `neutral-*`).
- Mantener consistencia con otros componentes de la app.

### Diseño
- El componente debe tener fondo blanco o según Design System.
- Bordes redondeados usando tokens (`rounded-lg`).
- Sombras sutiles para dar profundidad.
- El botón de búsqueda debe destacar visualmente.

### Dependencias / Paquetes
- No agregar dependencias nuevas salvo que esta HU lo indique explícitamente.
- Si aparece error 403/conectividad:
  1) Detenerse
  2) Pedir dominios requeridos
  3) Explicar por qué se requieren
  4) No continuar

## Implementación sugerida

### Estructura del componente:

El componente de búsqueda/filtro debe incluir:
- **Filtro de tipo**: Dropdown o botón con opciones (Perros, Gatos, etc.)
- **Filtro de ubicación**: Dropdown o botón con opciones de ubicación
- **Botón de búsqueda**: Botón destacado para ejecutar la búsqueda

### Mejoras visuales sugeridas:

1. **Contenedor principal**:
   - Fondo blanco con sombra sutil
   - Bordes redondeados
   - Padding apropiado
   - Espaciado entre elementos

2. **Filtros**:
   - Estilo consistente entre ambos
   - Iconos alineados correctamente
   - Estados hover y focus visibles
   - Texto legible y bien espaciado

3. **Botón de búsqueda**:
   - Color destacado (usar `brand-500`)
   - Tamaño apropiado
   - Estado hover con efecto
   - Icono si aplica

4. **Layout**:
   - En desktop: elementos en fila horizontal
   - En móvil: elementos apilados verticalmente
   - Espaciado consistente

### Archivos a revisar/modificar:

- Componente de búsqueda/filtro (ubicación a determinar según estructura actual)
- Posiblemente en `src/presentation/components/home/` o similar
